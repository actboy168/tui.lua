# Roadmap 编写规范

## Stage 命名

Stage 标题用递增整数（Stage 6, Stage 7...），不加字母后缀（5A / 5B / 5C）。

**Why:** 5A/5B/5C 当时是为了拆分"测试增强"大块，但效果是 roadmap 连续 Stage 号被占掉、阅读起来比平铺更乱。

**How to apply:** 每开始新阶段就递增整数。已完成的 Stage 5A/5B 保留历史名（已入 git），不追溯改名。

## Stage 颗粒度

一个 stage = 一组相关能力收尾，不是一个组件/一个 hook 等于一个 stage。

**Why:** 单件功能太细碎，stage 号膨胀快、changelog 噪声大。

**How to apply:**
- 开新 stage 前先问"这组工作有没有更大的外部主题"；有就用主题命名
- 同一 stage 下 bullet 可以多，但 stage 号不要膨胀
- 已命名的 stage 允许追加 bullet（如 Stage 19 — Select 扩成"AI chat 三件套收尾"），不需开新 stage
- 单件"顺手 fix"或纯架构改进直接吸附到同期 stage 的副作用里，不单独立 stage

## 简化方案的待办写法

采纳"够用但不完善"方案时，完善方案的记录方式：

1. **位置**：roadmap 的**未完成 · 按类别**下（作为独立 bullet），不是已完成条目的子清单
2. **措辞**：面向未来要做什么（"支持…"、"改为…"），不要括注"(Stage N 简化)"这类来源标记

**Why:** 待办项写"谁的遗留"会让未完成列表越读越像历史记录而非可执行计划。如确需追溯来源，靠 git log / git blame。

**How to apply:** 已完成条目只写做了什么，不列"简化点"清单；待做 bullet 直接写目标行为 + 当前状态。

## 已完成特性的记录

已支持的特性通过代码和 LuaDoc 注释表达，不再维护单独的 features.md 清单。

**How to apply:**
- 新增组件/Hook 时，在代码中写 LuaDoc 注释（---@param / ---@return）
- 外部可见的 API 通过 `tui.*` 模块导出，内部私有字段以下划线开头
- 不写测试相关 bullet：测试是内部质量手段，不进特性清单
- Roadmap 的正在进行/未完成区块不受此约束，可以写细
- 实现细节和设计决策放到 `docs/decisions/` 里

## Stage 决策记录标准

每做完一个 stage 不要机械地写决策记录。先过一遍以下清单，能留下的才写：

- ✅ 写：**非显然的选择理由** — "为什么不是方案 X"，否决的备选方案
- ✅ 写：**代码里看不出来的外部约束** — 平台、工具链、上游库限制
- ✅ 写：**会影响未来决策的判断性结论** — 如"阈值取 N 是对齐某先例"
- ❌ 不写：功能清单（"实现了 useMemo / useCallback"）— 代码和 LuaDoc 已有
- ❌ 不写：内部算法、函数名、文件行号 — 读代码更准
- ❌ 不写：bug 修复经过 — git log / commit message 有

已有的决策记录留着，除非后续 stage 改动让内容过时才清理/更新。
